# CodeGraph — Rust 复现开发计划

> 基于 GitNexus（TypeScript）的思想精髓，用 Rust 重新设计和实现，不照搬原代码。

---

## 设计哲学差异

| 维度 | GitNexus (TypeScript) | CodeGraph (Rust) |
|------|----------------------|-------------------|
| 运行时 | Node.js 事件循环 | tokio 异步运行时 |
| 内存管理 | GC + Worker 线程 | 所有权 + 零拷贝 |
| 图存储 | 外部图数据库 | InMemoryGraphStore + JSONL 持久化 |
| AST 解析 | tree-sitter WASM/原生绑定 | tree-sitter 原生绑定（tree-sitter crate） |
| 搜索 | 自实现 BM25 + transformers.js | Tantivy 全文检索 + 可选向量索引 |
| 嵌入 | HuggingFace transformers.js (JS) | ONNX Runtime / Candle（纯 Rust 推理） |
| 聚类 | Graphology Leiden | petgraph + 自实现 Leiden |
| CLI | Commander.js | clap v4 |
| MCP | 自实现 stdio 传输 | rmcp crate |
| 并发 | Worker 线程 + async | rayon 并行 + tokio 异步 |
| 增量索引 | 规划中（未实现） | **从第一天设计为增量优先** |

## 第一阶段：项目骨架与核心数据模型

### 1.1 项目初始化与 workspace 搭建

```
codegraph/
├── Cargo.toml              # workspace 根
├── crates/
│   ├── cg-core/            # 核心数据模型、图结构
│   ├── cg-parser/          # AST 解析管线
│   ├── cg-graph/           # 图存储引擎
│   ├── cg-search/          # 混合搜索引擎
│   ├── cg-embed/           # 嵌入生成
│   ├── cg-mcp/             # MCP 服务器
│   ├── cg-cli/             # CLI 入口
│   ├── cg-server/          # HTTP API 服务
│   └── cg-common/          # 共享类型、工具函数
├── fixtures/               # 测试用例代码片段
└── tests/                  # 集成测试
```

- 使用 Cargo workspace 管理多 crate
- 每个 crate 可独立编译测试
- `cg-common` 定义所有共享类型（类似 gitnexus-shared 的角色，但用 Rust trait/enum 重新设计）

### 1.2 核心类型系统设计 (`cg-common`)

- **`NodeKind` 枚举**：用 Rust enum 替代原来的字符串字面量节点类型，编译期穷举检查

  ```rust
  enum NodeKind {
      File, Folder, Function, Class, Method, Interface,
      Struct, Enum, Trait, Impl, Const, Module, Route, ...
  }
  ```

- **`EdgeKind` 枚举**：同样编译期安全

  ```rust
  enum EdgeKind {
      Contains, Calls, Imports, Extends, Implements,
      MemberOf, StepInProcess, HandlesRoute, ...
  }
  ```

- **`CodeNode` 结构体**：统一节点表示，用 `#[non_exhaustive]` 保持前向兼容
- **`CodeEdge` 结构体**：带置信度 `f64`、原因 `String`、可选步骤号
- **`Language` 枚举**：支持的语言列表
- **`Confidence` 类型别名**：`f64`，附带常量（同文件 0.95、导入作用域 0.9、全局 0.5）
- **`ImportSemantics` 枚举**：`Named`、`WildcardLeaf`、`WildcardTransitive`、`Namespace`
- **`MroStrategy` 枚举**：`FirstWins`、`C3`、`RubyMixin`、`None`

### 1.3 知识图谱内存模型 (`cg-core`)

- **`KnowledgeGraph` 结构体**：
  - `nodes: DashMap<NodeId, CodeNode>` — 并发安全节点存储
  - `out_edges: DashMap<NodeId, Vec<CodeEdge>>` — 出边邻接表
  - `in_edges: DashMap<NodeId, Vec<NodeId>>` — 入边反向索引
  - `symbol_index: DashMap<(String, Language), Vec<NodeId>>` — 名称→节点索引
  - `file_index: DashMap<PathBuf, Vec<NodeId>>` — 文件路径→节点索引
- **`NodeId`**：newtype 包装 `u64`，用 FxHash 计算，避免字符串 ID 开销
- **增量更新支持**：每个方法提供 `apply_delta(Delta)` 接口，从第一天就支持增量

## 第二阶段：文件扫描与目录结构解析

### 2.1 文件系统遍历 (`cg-parser`)

- **`FileScanner`**：基于 `ignore` crate（ripgrep 同款），支持 `.gitignore` 规则
  - 输出：`Vec<FileInfo>` — 路径、大小、语言推断
  - 并行扫描：rayon 并行遍历目录
  - 语言推断：基于扩展名 → `Language` 映射表

### 2.2 目录结构构建

- **`StructureBuilder`**：从 `FileInfo` 列表构建 Folder/File 层级
  - 生成 CONTAINS 边
  - 构建路径集合 `HashSet<PathBuf>` 用于后续导入解析
  - 输出类型：`StructureOutput { all_paths, folder_nodes, file_nodes }`

### 2.3 Markdown 解析

- **`MarkdownParser`**：基于 `pulldown-cmark` crate
  - 提取标题层级（Section 节点）
  - 检测交叉引用链接
  - 生成 CONTAINS / 引用边

## 第三阶段：Tree-sitter 多语言 AST 解析（核心）

### 3.1 Tree-sitter 基础设施

- 使用 `tree-sitter` crate + 各语言的 `tree-sitter-xxx` crate
- **`ParserPool`**：基于 `rayon::ThreadPool` 的并行解析池
  - 每个线程拥有独立的 `Parser` 实例（tree-sitter Parser 非线程安全）
  - 按 ~20MB 字节预算分块处理文件（与原项目相同的内存约束思想）
- **`GrammarRegistry`**：按需加载语言语法，缓存已加载的 `Language` 实例

### 3.2 统一捕获标签系统

- **`CaptureTag` 枚举**：对应原项目的统一标签概念

  ```rust
  enum CaptureTag {
      DefinitionClass,
      DefinitionFunction,
      DefinitionMethod,
      CallName,
      ImportSource,
      HeritageExtends,
      HeritageImplements,
      // ...
  }
  ```

- 每种语言将自己的 tree-sitter 查询结果映射到统一的 `CaptureTag`
- 下游处理无需关心语言差异

### 3.3 Language Provider trait 设计

- **`LanguageProvider` trait**：

  ```rust
  trait LanguageProvider: Send + Sync {
      fn id(&self) -> Language;
      fn extensions(&self) -> &[&str];
      fn tree_sitter_queries(&self) -> &str;         // S-expression 查询
      fn import_semantics(&self) -> ImportSemantics;
      fn resolve_import(&self, target: &str, from: &Path, all_files: &HashSet<PathBuf>) -> Option<PathBuf>;
      fn is_exported(&self, name: &str, node: &Node, source: &[u8]) -> bool;
      fn type_config(&self) -> TypeConfig;
      fn mro_strategy(&self) -> MroStrategy;
  }
  ```

- 用 trait object `Box<dyn LanguageProvider>` 替代原项目的 Record 类型
- 每个 provider 是一个独立模块，零耦合

### 3.4 符号提取器（Extractor 模式）

每种提取器用 trait 定义接口，各语言通过配置组合：

- **`SymbolExtractor` trait**：提取函数/类/方法/接口
- **`CallExtractor` trait**：提取调用关系
- **`ImportExtractor` trait**：提取导入语句
- **`HeritageExtractor` trait**：提取继承/实现关系
- **`FieldExtractor` trait**：提取字段/属性
- **`VariableExtractor` trait**：提取变量/常量
- **`TypeExtractor` trait**：提取类型注解

每种提取器返回统一的 `Vec<ExtractedSymbol>` / `Vec<ExtractedCall>` 等，语言无关。

### 3.5 首批语言支持（按优先级）

1. **Rust** — 优先支持自身
2. **TypeScript/JavaScript** — 最大用户群体
3. **Python** — 最流行的动态语言
4. **Go** — 简单的导入模型，验证 wildcard-leaf 语义
5. **Java** — 验证 named 导入 + C3/first-wins MRO
6. **C/C++** — 验证 wildcard-transitive (#include 链)
7. **C#** — 验证 scope-resolution 管线

后续：Kotlin, PHP, Ruby, Swift, Dart, COBOL(正则), Markdown

## 第四阶段：解析管线（Pipeline）编排

### 4.1 DAG 管线框架

- **`PipelinePhase` trait**：

  ```rust
  trait PipelinePhase: Send + Sync {
      fn name(&self) -> &str;
      fn deps(&self) -> &[&str];
      fn execute(&self, ctx: &mut PipelineContext, deps: &PhaseResults) -> Result<PhaseOutput>;
  }
  ```

- **`PipelineRunner`**：
  - Kahn 拓扑排序验证 DAG（检测循环、缺失依赖）
  - 按拓扑序顺序执行，无依赖的阶段可并行（`tokio::join!` / `rayon::scope`）
  - 每个 phase 只接收声明依赖的输出（防止隐式耦合）
  - 错误包装：包含阶段名、耗时、受影响的下游阶段数

### 4.2 十二阶段实现

| 阶段 | 实现 crate | 关键依赖 | 输出 |
|------|-----------|---------|------|
| **scan** | cg-parser | ignore crate | `Vec<FileInfo>` |
| **structure** | cg-parser | scan | Folder/File 节点 + CONTAINS 边 |
| **markdown** | cg-parser | structure | Section 节点 + 交叉引用边 |
| **cobol** | cg-parser | structure | COBOL 段节点（正则） |
| **parse** | cg-parser | structure+markdown+cobol | 符号节点 + IMPORTS/CALLS/EXTENDS 边 |
| **routes** | cg-parser | parse | Route 节点 + HANDLES_ROUTE 边 |
| **tools** | cg-parser | parse | Tool 节点 + HANDLES_TOOL 边 |
| **orm** | cg-parser | parse | QUERIES 边 |
| **crossFile** | cg-core | parse+routes+tools+orm | 跨文件类型传播 |
| **mro** | cg-core | crossFile+structure | METHOD_OVERRIDES + METHOD_IMPLEMENTS 边 |
| **communities** | cg-core | mro+structure | Community 节点 + MEMBER_OF 边 |
| **processes** | cg-core | communities+routes+tools+structure | Process 节点 + STEP_IN_PROCESS 边 |

## 第五阶段：调用解析与作用域解析

### 5.1 调用解析管线（Call Resolution Pipeline）

6 阶段流水线，与原项目思想相同但实现不同：

1. **extract-call** — 从 AST 提取调用点（`ExtractedCallSite`）
2. **classify-form** — 分类调用形式（`Free` / `Member` / `Constructor`）
3. **infer-receiver** — 推断接收者类型（支持 `infer_implicit_receiver` hook）
4. **select-dispatch** — 选择分派策略（支持 `select_dispatch` hook）
5. **resolve-target** — MRO 遍历查找目标方法
6. **emit-edge** — 写入 CALLS 边到知识图谱

每种语言可在 3-4 阶段插入自定义逻辑（通过 trait 方法）。

### 5.2 三层名称查找算法

- **Tier 1（同文件，置信度 0.95）**：符号表直接查找
- **Tier 2（导入作用域，置信度 0.9）**：
  - Named：沿 `import { X }` 链追踪
  - WildcardLeaf：导入包的全部导出符号
  - WildcardTransitive：`#include` 闭包链
  - Namespace：模块别名解析
- **Tier 3（全局，置信度 0.5）**：O(1) 索引查找（类名、接口名、可调用符号）

### 5.3 作用域解析管线（Scope Resolution，新架构）

- **`ScopeResolver` trait**：

  ```rust
  trait ScopeResolver: Send + Sync {
      fn language_provider(&self) -> &dyn LanguageProvider;
      fn populate_owners(&self, parsed: &mut ParsedFile);
      fn build_mro(&self, graph: &KnowledgeGraph, parsed: &ParsedFile) -> HashMap<DefId, Vec<DefId>>;
      fn resolve_import_target(&self, target: &str, from: &Path, all: &HashSet<PathBuf>) -> Option<PathBuf>;
      fn merge_bindings(&self, existing: &Binding, incoming: &Binding, scope_id: ScopeId) -> MergeResult;
      fn arity_compatibility(&self, caller_arity: usize, target_arity: usize) -> bool;
  }
  ```

- 管线阶段：`extractParsedFile → finalizeScopeModel → resolveReferenceSites → emitEdges`
- 从第一天就作为默认路径（不再保留 legacy 调用解析 DAG 作为备选）

### 5.4 MRO（方法解析序）实现

- **C3 线性化**：纯算法实现，参考 Python C3 规范
- **First-wins**：按声明顺序第一个匹配
- **Ruby-mixin**：深度优先 + 单调性检查
- 统一入口：`resolve_method(owner: NodeId, method_name: &str, strategy: MroStrategy) -> Option<NodeId>`

## 第六阶段：图存储引擎（内存图 + JSONL）

### 6.1 设计选择

CodeGraph 采用纯 Rust 内存图作为默认存储后端，原因如下：

1. **零外部依赖**：不需要 CMake、动态库或嵌入式数据库运行时，单二进制即可分发
2. **构建简单**：`cargo build` 即可，无跨平台 FFI 问题
3. **足够快**：`DashMap` + `FxHash` 的并发图结构足以支撑中小仓库的索引和查询
4. **易于增量**：JSONL 行存储天然适合基于内容哈希的增量更新
5. **可扩展**：`GraphStore` trait 保留，未来仍可接入其他持久化后端

### 6.2 存储适配层 (`cg-graph`)

- **`GraphStore` trait**：统一的图操作接口

  ```rust
  trait GraphStore: Send + Sync {
      fn add_node(&self, node: CodeNode) -> Result<()>;
      fn add_edge(&self, edge: CodeEdge) -> Result<()>;
      fn get_node(&self, id: &str) -> Option<CodeNode>;
      fn query_neighbors(&self, id: &str, direction: Direction, edge_kind: Option<EdgeKind>) -> Vec<CodeEdge>;
      fn query_by_label(&self, label: NodeKind) -> Vec<CodeNode>;
      fn all_nodes(&self) -> Vec<CodeNode>;
      fn batch_insert_nodes(&self, nodes: Vec<CodeNode>) -> Result<()>;
      fn batch_insert_edges(&self, edges: Vec<CodeEdge>) -> Result<()>;
  }
  ```

- **`InMemoryGraphStore` 实现**：管线执行期间的内存图
  - 管线各阶段在内存中构建 `KnowledgeGraph`（DashMap）
  - 支持 `export_jsonl` / `import_jsonl` 持久化
  - 支持 `export_csv` 批量导出（便于未来接入外部数据库）
  - 保留内存图用于管线内部的快速查询（MRO 遍历、社区检测等）

### 6.3 图模型

图模型由 `NodeKind` / `EdgeKind` 枚举定义：

- **节点类型**：File, Folder, Function, Class, Interface, Method, Struct, Enum, Trait, Impl, Const, Module, Route, Tool, Community, Process, Section 等
- **关系类型**：Contains, Calls, Imports, Extends, Implements, MemberOf, StepInProcess, HandlesRoute, HandlesTool, Queries, MethodOverrides, MethodImplements 等
- **属性**：每个节点/边携带 `confidence`（置信度）、`reason`（原因）、`step`（可选步骤号）

### 6.4 存储目录结构

```
<repo>/.codegraph/
├── meta.json            # lastCommit, indexedAt, stats
├── nodes.jsonl          # 所有节点（每行一个 JSON）
├── edges.jsonl          # 所有边（每行一个 JSON）
└── hashes.json          # 文件内容哈希缓存

~/.codegraph/
└── registry.json        # 全局仓库注册表
```

### 6.5 查询方式

- **内存遍历**：直接通过 `KnowledgeGraph` 的邻接表做 BFS/DFS
- **结构化搜索**：`GraphStore::query_by_label`、`query_neighbors`、`all_nodes`
- **全文搜索**：由 `cg-search` 的 Tantivy 索引提供（见第七阶段）
- **Cypher**：当前版本不支持 Cypher 查询语言

## 第七阶段：混合搜索引擎

### 7.1 全文检索（Tantivy）

**方案：Tantivy 全文检索**

- 纯 Rust 实现，无外部数据库依赖
- 索引字段：`name`（BM25）、`file_path`、`content`、`kind`（facet）、`language`（facet）
- 支持内存模式和磁盘模式
- 适用于小到中型仓库

### 7.2 语义向量搜索

- **向量索引**：当前由 `cg-embed` 在内存中计算余弦相似度，未来可接入 HNSW 库
- **嵌入生成**：
  - 方案 A：调用远程 API（OpenAI / 自部署 embedding 服务）
  - 方案 B：使用 `candle` crate 本地推理
  - 方案 C：使用 `ort` (ONNX Runtime) 本地推理
  - 默认使用 Dummy embedder，便于快速测试
- **文本分块**：基于 AST 结构的分块（函数体、类体），而非纯字符分块
- **内容哈希增量**：基于 SHA-256 内容哈希跳过未变更节点的嵌入生成

### 7.3 混合融合排序

- **RRF（Reciprocal Rank Fusion）**：K=60，与原项目相同

  ```rust
  fn rrf_merge(bm25_results: Vec<SearchHit>, vector_results: Vec<SearchHit>, k: usize) -> Vec<SearchHit>
  ```

- 支持按过程（Process）分组展示搜索结果

## 第八阶段：社区检测与执行流追踪

### 8.1 Leiden 社区检测

- 基于 `petgraph` 构建代码关系子图
- 自实现 Leiden 算法（模块度优化 + 细化阶段）
- 输出：`Vec<Community>`，每个社区包含：
  - 启发式标签（从成员的高频词提取）
  - 内聚度评分
  - 成员节点列表
- 生成 MEMBER_OF 边

### 8.2 执行流追踪（Process Detection）

- **入口点评分**：
  - 框架检测（Next.js handler、Express route、Go main、Rust main 等）
  - AST 模式匹配 + 框架乘数
  - 输出：入口点节点 + `entry_point_score` + `entry_point_reason`
- **调用链追踪**：从入口点出发，沿 CALLS 边 BFS/DFS 遍历
- **过程分类**：
  - `IntraCommunity`：所有步骤在同一社区
  - `CrossCommunity`：跨社区调用链
- 输出：Process 节点 + STEP_IN_PROCESS 边

## 第九阶段：MCP 服务器与工具

### 9.1 MCP 服务器实现

- 使用 `rmcp` crate 或自实现 MCP stdio 传输
- **`McpServer`**：
  - 读取 `~/.codegraph/registry.json` 发现所有已索引仓库
  - 懒加载图存储连接
  - 连接池：最多 5 个并发连接，5 分钟空闲回收

### 9.2 工具实现

| 工具 | 功能 | 核心实现 |
|------|------|---------|
| **list_repos** | 发现所有已索引仓库 | 读 registry.json |
| **query** | 混合搜索 | Tantivy BM25 + 内存搜索 + 可选向量 RRF |
| **context** | 360 度符号视图 | 图遍历（入边/出边/过程） |
| **impact** | 爆炸半径分析 | BFS 上/下游遍历 + 深度分组 + 风险摘要 |

| **api_impact** | API 变更影响报告 | 路由→处理器→调用链遍历 |

## 第十一阶段：HTTP API 与 Web 桥接

### 11.1 HTTP API 服务器

- 使用 `axum` 框架
- 端点设计：
  - `GET /api/repos` — 列出仓库

## 第十二阶段：增量索引（核心差异化特性）

**这是原项目尚未实现的关键特性，Rust 版从第一天设计为增量优先。**

### 12.1 基于 Git Diff 的增量更新

- 使用 `git2` crate 获取自上次索引以来的变更文件列表
- 只重新解析变更的文件
- 增量图更新：
  - 删除旧文件相关的所有节点和边
  - 重新解析新文件并添加节点和边
  - 重新运行受影响的后续阶段（crossFile, mro, communities, processes）

### 12.2 内容哈希缓存

- 每个文件的 SHA-256 内容哈希
- 下次索引时跳过未变更文件
- 嵌入向量同样基于内容哈希增量更新

### 12.3 过期检测

- 比较索引记录的 `lastCommit` 与当前 `HEAD`
- 输出过期提示：`"索引已落后 5 个提交，建议运行 codegraph analyze 更新"`

### 13.1 组配置

- **HTTP 模式提取**：从路由定义提取 API 契约
- **消息队列模式提取**：从发布/订阅代码提取主题契约
- **契约匹配**：跨仓库匹配 provider 和 consumer
- **桥接图**：跨仓库的 CONTRACT_PROVIDES / CONTRACT_CONSUMES 边

### 13.3 跨仓库影响分析

- 从一个仓库的本地影响出发
- 通过桥接图发现跨仓库的级联影响

## 第十五阶段：测试体系

### 15.1 测试分层

- **单元测试**：每个 crate 内 `#[cfg(test)] mod tests`
- **集成测试**：`tests/` 目录，跨 crate 测试
- **Fixture 测试**：每种语言的代码片段 → 期望的图结构
  - `fixtures/rust/basic_struct.rs` → 期望提取出 Struct 节点 + impl 方法
  - `fixtures/typescript/import_chain.ts` → 期望解析出跨文件导入
- **快照测试**：使用 `insta` crate 对图结构做快照对比
- **性能基准**：使用 `criterion` crate 对关键路径基准测试

### 15.2 管线一致性验证

- 类似原项目的 scope-parity 检测：
  - 对每种语言，分别用 legacy 路径和 scope-resolution 路径解析
  - 比较输出的图结构是否一致
  - 不一致时 CI 失败

### 15.3 测试覆盖率

- 使用 `cargo-llvm-cov` 生成覆盖率报告
- 目标：核心管线 ≥ 80% 行覆盖率

## 与原项目的关键差异总结

1. **图存储**：使用 InMemoryGraphStore + JSONL，零外部数据库依赖，单二进制即可运行
2. **搜索**：Tantivy 全文检索 + 可选向量相似度，纯 Rust 实现
3. **Cypher**：当前版本不支持 Cypher 查询语言
4. **增量索引**：从第一天设计为增量优先，基于 JSONL 和内容哈希
5. **类型安全**：用 Rust enum 代替字符串字面量，编译期穷举检查所有节点/边类型
6. **作用域解析**：从第一天就用 scope-resolution 作为默认路径，不保留 legacy 调用解析 DAG
7. **单二进制**：最终产物是一个零依赖的可执行文件，不需要 Node.js 运行时
